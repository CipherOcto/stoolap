# Hexary Merkle Proof Design

**Date:** 2025-02-28
**Status:** Design
**Author:** AI Design Assistant

## Problem Statement

The current `MerkleProof` type is designed for binary trees (left/right branching), but `RowTrie` is a 16-way hexary trie. This architectural mismatch means:
- Proofs cannot be verified
- The efficient hexary trie structure cannot be properly utilized
- Proof sizes are larger than necessary

## Design Goals

1. **Compact proofs** - Minimize proof size to reduce transaction costs
2. **Efficient verification** - Fast single-proof and batch-proof verification
3. **Extensible** - Support for future trie enhancements
4. **Clean replacement** - Drop-in replacement for `MerkleProof`

## Architecture

### Core Data Structures

```rust
/// Hexary Merkle proof for 16-way trie verification
pub struct HexaryProof {
    /// Hash of the value being proven (row hash)
    pub value_hash: [u8; 32],

    /// Proof levels from root to leaf
    pub levels: Vec<ProofLevel>,

    /// Expected Merkle root
    pub root: [u8; 32],

    /// Nibble path (2 nibbles packed per byte, LSB first)
    pub path: Vec<u8>,
}

/// One level of a hexary proof
pub struct ProofLevel {
    /// 16-bit bitmap indicating which children have hashes
    pub bitmap: u16,

    /// Sibling hashes (only non-empty children)
    pub siblings: Vec<[u8; 32]>,
}
```

### Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Path encoding | Compact nibble vector | 2 nibbles per byte for space efficiency |
| Sibling encoding | Bitmap-based | Extensible, handles sparse tries well |
| Extension nodes | Flatten during proof generation | Simplifies verification |
| Verification | Streaming | Fast rejection of invalid proofs |
| Batch verification | Parallel + optimized | Maximum throughput for blocks |
| Serialization | Solana-style | Battle-tested, zero-copy reads |

### Nibble Path Encoding

- Path `[5, 12, 3]` encodes as bytes `[0x35, 0xC3]`
- Odd paths like `[5, 12, 3, 7]` encode as `[0x35, 0xC3, 0x07]`
- Each nibble indicates which of 16 children to take at that level

### Bitmap Encoding

At each branch level:
- 16-bit bitmap indicates which child positions have hashes
- Only sibling hashes are stored (not the path child)
- Verification reconstructs all 16 children from bitmap + siblings + path

Example: Taking child 5, with children 3 and 12 also existing
- `bitmap = 0b1000000000001000` (bits 3, 5, 12 set)
- `siblings = [hash_3, hash_12]`

## Proof Generation

Proofs are generated by walking the trie from root to target row:

1. Start at root, track path nibbles taken
2. At each Branch node:
   - Record bitmap of existing children
   - Collect sibling hashes (exclude path child)
   - Continue down path
3. At Extension node:
   - Flatten: add prefix nibbles directly to path
   - Continue to child (no proof level added)
4. At Leaf node:
   - Return row hash as value hash

## Verification

Streaming verification walks from leaf to root:

1. Start with `value_hash`
2. For each level (leaf to root):
   - Reconstruct all 16 children from bitmap + siblings + current_hash
   - Hash all 16 children to get parent hash
   - Continue with parent as current_hash
3. Final hash must equal `root`

Fails immediately on any mismatch.

## Batch Verification

Three modes supported:

1. **Parallel** - Use rayon to verify multiple proofs across CPU cores
2. **Optimized** - Amortize SHA-256 computations across proofs
3. **Combined** - Parallel threads with cryptographic batching

## Serialization Format

Solana-style binary format:

| Field | Size | Description |
|-------|------|-------------|
| value_hash | 32 bytes | Hash of proven value |
| root | 32 bytes | Expected Merkle root |
| path_len | 1 byte | Length of path |
| path | variable | Packed nibble path |
| levels_len | 1 byte | Number of levels |
| levels | variable | Proof levels |

Each level:
| Field | Size | Description |
|-------|------|-------------|
| bitmap | 2 bytes | 16-bit child bitmap |
| sibling_count | 1 byte | Number of siblings |
| siblings | variable | Sibling hashes (32 bytes each) |
